# 数据库结构说明

这份文档用来帮助理解当前项目里的 SQLite 数据库结构。

项目里的数据库文件放在：

```text
myapp/databases/
```

里面有很多 `.db` 文件。一般来说，一个 `.db` 文件是一份课程数据快照，文件名里会带学制、学期和生成时间，例如：

```text
UGRD-2268-26FALL-2026-05-05-12-51-13.db
GRAD-2268-26FALL-2026-05-01-16-54-55.db
course_assistant.db
```

其中：

- `UGRD` 表示本科。
- `GRAD` 表示研究生。
- `2268` 是学期代码。
- `26FALL` 是更容易读的学期标签。
- 最后的日期时间表示数据库生成时间。

当前默认配置在 `config.py` 里：

```text
DEFAULT_TERM = "2268"
DEFAULT_CAREER = "UGRD"
```

也就是说，默认看的是 2026 Fall 本科课程数据。

## 总体结构

当前数据库主要有两张表：

```text
courses
metadata
```

`courses` 是核心表，保存课程和 section 信息。

`metadata` 是辅助表，保存同步接口时的一些额外信息。

## courses 表

`courses` 表里，一行通常代表一个课程 section。

比如同一门课如果有 `001`、`002` 两个 section，通常会有两行。

### 主键和唯一标识

```text
id
course_key
```

`id` 是数据库自动生成的数字主键。

`course_key` 是项目自己生成的唯一 key，用来判断一条课程记录是否已经存在。它有唯一约束，不能重复。

### 学期和课程基本信息

```text
term
subject
subject_descr
catalog_nbr
class_section
course_name
class_nbr
```

含义：

- `term`：学期代码，例如 `2268`。
- `subject`：学科代码，例如 `COMPSCI`、`MATH`。
- `subject_descr`：学科名称或描述。
- `catalog_nbr`：课程号，例如 `201`、`302`。
- `class_section`：section 编号，例如 `001`。
- `course_name`：课程标题。
- `class_nbr`：学校系统里的 class number，常用于查课程详情。

如果要显示一门课的代码，一般是：

```text
subject + " " + catalog_nbr
```

例如：

```text
COMPSCI 201
```

### 上课时间和教学信息

```text
meeting_duration
units
grading_basis
component
session_code
start_date
end_date
meeting_days
meeting_start_time
meeting_end_time
building
classroom
instructor_names
```

含义：

- `meeting_duration`：单次课时长。
- `units`：学分。
- `grading_basis`：评分方式。
- `component`：课程组成类型，例如 lecture、seminar、lab 等。
- `session_code`：session 代码。
- `start_date`：开始日期。
- `end_date`：结束日期。
- `meeting_days`：上课星期。
- `meeting_start_time`：上课开始时间。
- `meeting_end_time`：上课结束时间。
- `building`：上课教学楼，从大课搜索的 meeting 信息里解析得到。
- `classroom`：具体上课教室，通常来自大课搜索的 `facility_descr`，例如 `IB 2028 (Classroom)`。
- `instructor_names`：老师姓名，可能有多个老师。

### 选课容量和状态

```text
enrl_status
class_capacity
enrollment_available
wait_list_capacity
wait_list_available
```

含义：

- `enrl_status`：选课状态。
- `class_capacity`：课程容量。
- `enrollment_available`：剩余可选名额。
- `wait_list_capacity`：等待名单容量。
- `wait_list_available`：等待名单剩余名额。

### Combined section 信息

```text
is_combined
combined_sections_json
```

含义：

- `is_combined`：是否是 combined section，通常是 `Y` 或 `N`。
- `combined_sections_json`：combined section 的原始结构化信息，保存成 JSON 字符串。

### 大小课关联信息

```text
related_sections_json
```

含义：

- `related_sections_json`：这门课关联的大课(LEC)、实验(LAB)、习题课(REC)等关联信息，保存成 JSON 字符串。

数据来自 Shopping Cart 的 ClassDetails 接口 (`H_SHOPPING_CART.IScript_ClassDetails`)，保存的 JSON 包含两部分：

- `related_sections`：子课（LAB）视角，列出它必须/可选搭配的父课（LEC/SEM）和习题课（REC）。
- `enrollment_sections`：父课（LEC/SEM）视角，列出它可以选的子课（LAB）。

每个 `related_sections` 元素包含：

- `Component`：关联课类型，例如 LEC、LAB、REC。
- `Descr`：关联课类型描述。
- `IsRequired`：是否必须选（例如选了 LAB 的学生必须同时选对应的 LEC）。
- `Sections`：可选的 section 列表。

每个 `enrollment_sections` 元素是一个可选的子课，只存 `class_nbr` 和 `component`，展示时从本地数据库补全课程信息。

### Requisite 解析信息

```text
requisite_raw
requisite_parsed_json
requisite_tags
```

含义：

- `requisite_raw`：选课要求原始文本，来自 `detail_raw_json.section_info.enrollment_information.enroll_requirements`。
- `requisite_parsed_json`：由 `requisite_parser.py` 解析后的结构化结果，包含 `eval_tree`、`normalized_eval_tree`、`special_conditions` 等。
- `requisite_tags`：逗号分隔的分类标签，用于课程筛选。可能的值：`has_prereq`、`has_coreq`、`has_anti`、`has_grade`、`has_consent`、`has_unknown`。

### Reserved Seat 解析信息

```text
reserved_seats_raw
reserved_seats_parsed_json
reserved_seats_tags
```

含义：

- `reserved_seats_raw`：预留座位的原始 JSON（来自 `search_raw_json.reserve_caps` 或 `detail_raw_json.section_info.reserve_caps`）。
- `reserved_seats_parsed_json`：由 `reserved_seat_parser.py` 解析后的结构化结果，每个条件包含 `type`（`class_year`/`year_level`/`student_identity`/`unknown`）、`label`、`values`。
- `reserved_seats_tags`：逗号分隔的分类标签，用于课程筛选。可能的值：`has_class_year`、`has_year_level`、`has_identity`、`has_unknown`。

### 课程属性和毕业要求

```text
course_attributes
course_attribute_values
requirement_designation
```

含义：

- `course_attributes`：课程属性代码，多个值可能用逗号拼在一起。
- `course_attribute_values`：课程属性值，多个值可能用逗号拼在一起。
- `requirement_designation`：学校系统里的 requirement designation。

项目里筛选课程属性和 requirement designation 时，会用这些字段。

### 选课要求字段

```text
enroll_requirements
prereqs
antireqs
coreqs
```

含义：

- `enroll_requirements`：选课要求原文。
- `prereqs`：旧版先修课字段。
- `antireqs`：旧版反修课字段。
- `coreqs`：旧版同修课字段。

现在更完整、更新的数据一般在 `detail_raw_json` 里，所以新的 requisite parser 主要读原始 JSON，而不是只靠这些旧字段。

### 原始接口 JSON

```text
search_raw_json
detail_raw_json
enroll_detail_raw_json
search_synced_at
detail_synced_at
```

含义：

- `search_raw_json`：课程搜索接口返回的原始 JSON。
- `detail_raw_json`：课程详情接口返回的原始 JSON。
- `enroll_detail_raw_json`：Shopping Cart 选课详情接口返回的原始 JSON（包含 `related_sections`、`enrollment_sections` 等大小课关系数据）。
- `search_synced_at`：搜索接口数据同步时间。
- `detail_synced_at`：详情接口数据同步时间。

这些 JSON 字段很重要，因为很多学校接口里的细节没有被拆成单独列，而是原样保存在这里。requisite parser 和 reserved seat parser 都从这些原始 JSON 中提取数据。

## parser 主要读哪里

### Requisite parser

代码文件：`requisite_parser.py`

主要读取：

```text
detail_raw_json.section_info.enrollment_information.enroll_requirements
```

也就是课程详情 JSON 里的选课要求原文。

解析过程：
1. 提取 `enroll_requirements` 原文 → 存入 `requisite_raw`
2. 正则分词 → 构建 `eval_tree`（语法树）→ 规范化生成 `normalized_eval_tree`
3. 分类打标签 → 存入 `requisite_tags`（如 `has_prereq`、`has_grade` 等）

例如：

```text
Prerequisite: MATH 201 and COMPSCI 201; Anti-requisite: MATH 304
```

会被解析成包含 PRE（MATH 201 AND COMPSCI 201）和 ANTI（MATH 304）两个分支的树形结构。

详情页展示时，Eval Tree 和 Normalized Eval Tree 并排显示为彩色嵌套框框。

### Reserved seat parser

代码文件：`reserved_seat_parser.py`

主要读取：

```text
search_raw_json.reserve_caps
```

如果这里没有，再兜底读取：

```text
detail_raw_json.section_info.reserve_caps
```

解析过程：
1. 提取 `reserve_caps` 数组 → 存入 `reserved_seats_raw`
2. 对每个 cap 的描述文本做模式匹配（Class of XXXX / COXX / Y1-Y4 / international / chinese）
3. 分类打标签 → 存入 `reserved_seats_tags`

例如：

```text
Reserved for Class of 2026 stu
Reserved for International students
Reserved for Y1/Y2
```

会被解析成 `class_year`、`student_identity`、`year_level` 三种条件。

## metadata 表

`metadata` 表结构很简单：

```text
key
value
updated_at
```

含义：

- `key`：元数据名称，主键。
- `value`：JSON 字符串。
- `updated_at`：更新时间。

目前它主要用于保存搜索选项接口的同步信息，比如当前同步的是哪个 term、哪个 career、同步时间、接口返回内容等。

## 索引

`courses` 表上有这些索引：

```text
idx_courses_term
idx_courses_subject
idx_courses_class_nbr
idx_courses_instructor
idx_courses_building
```

分别用于加速这些字段的查询：

```text
term
subject
class_nbr
instructor_names
building
```

另外，`course_key` 有唯一约束，所以它也会有 SQLite 自动创建的唯一索引。

## 一个简单理解方式

可以把 `courses` 表分成两层：

第一层是常用筛选字段，已经拆成普通列：

```text
term
subject
catalog_nbr
course_name
meeting_days
meeting_start_time
building
classroom
instructor_names
class_capacity
enrollment_available
```

这些字段适合直接拿来搜索、筛选和展示。

第二层是原始 JSON 和解析结果：

```text
search_raw_json
detail_raw_json
enroll_detail_raw_json
related_sections_json
requisite_raw / requisite_parsed_json
reserved_seats_raw / reserved_seats_parsed_json
```

这些字段适合在需要更细信息时再解析，比如：

- 先修/同修/反修规则（requisite parser 从 `detail_raw_json` 提取原文，解析为 `eval_tree` 存进 `requisite_parsed_json`）
- reserved seats（`reserved_seat_parser` 从 `search_raw_json` / `detail_raw_json` 提取）
- combined section 的细节
- 大小课关联信息（`related_sections_json`）
- 学校接口里暂时还没有拆出来的字段

## 页面使用数据源的约定

课程筛选页默认使用“大课搜索”信息。

也就是说，课程列表里每一行显示的课程代码、课程名、课型、section、学分、session、上课时间、老师、评分方式、状态、剩余名额等，都应该优先来自：

```text
search_raw_json
```

如果 `search_raw_json` 里缺少某个字段，才退回使用 `courses` 表里已经拆出来的普通列。

详情页默认使用“课程详情”信息。

也就是说，详情页里的课程标题、授课方式、地点、容量、选课要求、reserved seats、combined sections、课程简介等，都应该优先来自：

```text
detail_raw_json
```

如果某门课还没有抓取过详情，或者 `detail_raw_json` 里没有对应字段，详情页再退回使用：

```text
search_raw_json
```

这样分工之后：

- 课程筛选页保持和大课搜索列表一致。
- 详情页尽量展示更完整的详细接口信息。
- 没有详情数据时，详情页也不会空掉，而是先展示大课搜索里已有的信息。

### 大小课关系数据来源

`related_sections_json` 的数据来自另一个独立接口 —— Shopping Cart 的 ClassDetails (`H_SHOPPING_CART.IScript_ClassDetails`)。这个接口需要 `X-CSRF-Token` 请求头（从 Cookie 里的 `CSRFCookie` 值获取），请求方式为 POST。

它只在"抓取详情"（`fetch_details=true`）或"刷新单个课程详情"时顺便抓取。如果该接口调用失败，不影响课程列表同步和普通详情抓取，只是这门课的 `related_sections_json` 会留空。
